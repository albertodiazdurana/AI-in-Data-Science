Friendly Introduction to SHAP and LIME: Why Explainability Matters
Let’s take a step back and talk about why model explainability is so important not just from a technical standpoint, but from a business and human perspective too.
What Problem Are We Trying to Solve?
Imagine building a model that predicts whether a loan should be approved. It has 95% accuracy — great! But what if a customer is denied a loan and asks, "Why?"
If all you can say is “because the model said so,” that’s not good enough not for your customer, your team, or your legal department.
You need to explain:
Which factors influenced the decision
How important each feature (or piece of information considered) was
Whether the prediction was fair and reliable
Explainability isn’t just about debugging (although it may help with this, too) it’s about trust, ethics, and effective communication.
Meet SHAP and LIME
We’ve talked in the previous lesson that there will be 2 tools that we’ll employ for explainability. Let’s talk about them now. These are SHAP and LIME. These two tools help us break down model predictions into understandable parts.
Let’s start talking about SHAP first!
SHAP (SHapley Additive exPlanations) is a tool that:
Is based on game theory (Shapley values).
Assigns each feature a “contribution score” for a prediction.
Works well with many model types (including tree-based models).
Helps answer: Which features pushed the prediction up or down?
On another hand there is LIME!
LIME (Local Interpretable Model-agnostic Explanations) is a solution that:
Approximates the behavior of a complex model with a simple, interpretable one but only locally around one prediction.
Helps answer: Why did the model make this specific decision?
Works well when you need to explain a single result clearly.
This brief but important intro is exactly what we need at this stage! Later on we’ll jump into each tool and show how it works end to end. Right now, just know that they exist! More importantly, we need to talk at this stage when these tools come handy!
When Would You Use These Tools?
The sky is the limit but the common cases include scenarios:
When stakeholders ask for justification behind predictions.
When you want to verify your model isn’t relying on biased or irrelevant features.
When you're deploying models in regulated industries (like healthcare or finance).
When debugging misclassifications or strange model behavior.
So, with this in mind we’ll move on and dive into these tools! In the next lessons, we’ll go hands-on with both SHAP and LIME. You’ll see how to apply them to real models and visualize the insights they provide, so you can use them to create clearer, more trustworthy reporting.
AI can help us move fast but explainability helps us move responsibly!

In the next step you’ll see why we need these SHAP values because we’ll visualize them, but for now lets talk about are other explainers that are available for us in SHAP to get SHAP values.
We’ve just used the TreeExplainer which is suitable for tree-based models like:
xgboost.XGBClassifier, xgboost.XGBRegressor
lightgbm.LGBMClassifier, LGBMRegressor
sklearn.ensemble.RandomForestClassifier, RandomForestRegressor
CatBoost models
It is very fast and accurate thanks to algorithmic optimizations for tree structures.
As an alternative, there is a LinearExplainer which can be used for linear models like:
sklearn.linear_model.LinearRegression, LogisticRegressionAor any model that has a linear decision function
Note: Works best when features are independent. If features are correlated, it may not produce intuitive explanations unless you specify a background dataset and set feature_perturbation="interventional".
The third candidate to consider would be DeepExplainer which us used for deep learning models built withKeras or TensorFlow (i.e., models with differentiable operations)
Limitation: Only supports certain activation functions and simple network architectures.
and last but not least we need to talk about KernelExplainer that is a universal option, meaning that it can be used with mny model (black-box explainer). To be precise it:
Works with sklearn, XGBoost, Keras, PyTorch, or custom models
Can be used even when you don’t have access to model internals
But it all comes with a downside: It is going to be way slower and less scalable!

Step 5: Local Explanation: One Row at a Time
What if we are interesting in explaining a single prediction? Can we do it? The answer is YES! Let’s now examine a single prediction to see how the model arrived at that specific result.
Copied code
to clipboard
12
# Explain a single prediction
shap.plots.waterfall(shap_values[0])
and here comes the output:
notion image
This line shows a waterfall plot that breaks down how the model arrived at the prediction for one specific row (in this case, the first test sample).
It answers the question: “Why did the model make this specific prediction?”
The waterfall plot shows three key parts:
Base Value:
This is the mean prediction of the model across all training data. In our case it is E[f(X)] = 154.38.
SHAP Values (Feature Contributions):
Each feature either adds to or subtracts from this base value. For example:
Copied code
to clipboard
1234
-12.37 from bmi
+2.14 from age
-2.23 from s6
-5.31 from bp
Model Output (Final Prediction):
The final prediction (f(x)) is calculated as:
Copied code
to clipboard
1
154.35 (base) - 12.37 + 2.14 - 2.23 - 5.31 - .... = 144
The plot visually builds this step-by-step:
It starts at the base value.
Bars going to the right show features that increased the prediction.
Bars going to the left show features that decreased it.
The end of the chart is the final prediction (e.g., 144).
What You’ve Learned
SHAP helps you go beyond accuracy by showing you how each feature influences model predictions. You can explain individual predictions or get global insights about the model’s logic.It works especially well with tree-based models like RandomForest or XGBoost: fast and accurate.

import pandas as pd
from sklearn.datasets import load_diabetes

Step 2: Load and Prepare the Dataset
We’ll use the Diabetes dataset from sklearn:
Copied code
to clipboard

data = load_diabetes()
X = pd.DataFrame(data.data, columns=data.feature_names)
y = data.target
We split the data into train/test sets and scale it. LIME perturbs input data around the instance we want to explain. Scaling ensures better behavior during this perturbation process.
Copied code
to clipboard
123456
X_train, X_test, y_train, y_test = train_test_split(X, y, random_state=42)

# Standardizing features improves LIME’s performance
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)
 
Step 3: Train a Predictive Model
Next, we train a Random Forest model which often delivers strong predictive performance but, as we know, may be difficult to interpret. That’s where LIME comes in.
Copied code
to clipboard
12
model = RandomForestRegressor(n_estimators=100, random_state=42)
model.fit(X_train_scaled, y_train)
Step 4: Initialize the LIME Explainer
Now let’s initializes the LimeTabularExplainer which is the right choice when working with structured/tabular data.
Copied code
to clipboard
1234567
explainer = LimeTabularExplainer(
    training_data=X_train_scaled,
    feature_names=X.columns.tolist(),
    mode='regression',
    verbose=True,
    random_state=42
)
There are other types of LIME explainers though:
LimeTextExplainer: For natural language/text classification.
LimeImageExplainer: For image classification tasks.
We choose LimeTabularExplainer because our features are structured and numeric and we’re solving a regression task. Now, briefly about the arguments:
training_data: Needed to simulate realistic perturbations.
feature_names: For readable output.
mode: 'regression' or 'classification'.
verbose=True: For debug-like detail during explanation.
Step 5: Explain a Prediction
Let’s explain why the model predicted a certain value for the first sample.
Copied code
to clipboard
12345678
sample_index = 0
sample = X_test_scaled[sample_index].reshape(1, -1)

explanation = explainer.explain_instance(
    data_row=X_test_scaled[sample_index],
    predict_fn=model.predict,
    num_features=5
)
Let’s briefly talk about the above code and what it has:
sample: The single instance we want to understand (a 1D array of standardized feature values).
Next, we call the expain_instance() method that has many parameters. Let’s discuss them:
data_row=sample: This is the input passed to LIME for local perturbation and explanation.
predict_fn=model.predict: Tells LIME how to get predictions.
num_features=5: Limits the output to the 5 most influential features.
As a result, under the hood LIME generates small variations of sample (perturbations). It passes these perturbed samples to the model (predict_fn). Then, it fits a simple, interpretable model (e.g., linear regression) on this local data. The output shows which features contributed most to the model's decision for this specific instance.

SHAP vs LIME
Did you spot any differences between SHAP and LIME so far? The most obvious one would be that LIME doesn’t has anything similar to SHAP values and can’t show the impact of features across the entire dataset. Other differences are neatly presented in the below table: 
Feature
LIME
SHAP
Accuracy
Approximate (uses a local linear model)
Theoretically consistent (Shapley values)
Use case
Fast, flexible, quick testing
More robust for trust in production
Visualizations
Simple and focused
Rich and detailed
Model support
Model-agnostic
Model-agnostic + optimized for tree-based
So, to make some conclusions, you can use LIME for fast and interpretable local explanations when you're experimenting or need simplicity. Consider using SHAP for more mathematically grounded, production-level model explanations, especially with tree models.
